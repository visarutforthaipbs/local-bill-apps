GOOGLE DRIVE SIGN-IN — EXACT PATH

Once Pro is active: Settings (gear) → "พื้นที่ทำงาน" (Workspace) → the button
"เชื่อมต่อ Google Drive" (Connect Google Drive). This button does not exist

Any Google account works. The requested scope is drive.file, which grants access

Note: on Google's consent screen, please make sure the Google Drive permission
checkbox is ticked before continuing. If it is left unticked, the app detects
that the permission was not granted and asks you to reconnect (it does not
enter a broken state).

ENTITLEMENTS — WHAT EACH ONE IS FOR AND WHERE TO SEE IT

- com.apple.security.network.server — the ONLY use is the loopback OAuth
  listener for "Connect Google Drive". We sign in using OAuth 2.0 with PKCE and
  a loopback redirect (RFC 8252 §7.3, "Loopback Interface Redirection"), which
  Google requires for installed desktop apps after deprecating the out-of-band
  method. On click, the app binds a temporary HTTP listener to 127.0.0.1 on an
  OS-assigned ephemeral port (listen(0, "127.0.0.1")), opens the browser to
  Google's consent screen with redirect_uri = http://127.0.0.1:<port>/callback,
  receives the authorization code on that loopback, validates the OAuth "state"
  parameter, exchanges the code for a token, and closes the listener
  immediately (a 5-minute timeout also force-closes it if sign-in is
  abandoned). It binds only to loopback — never 0.0.0.0 — accepts only the
  /callback path, is unreachable from outside the machine, and exists only for
  the seconds the user is signing in. BillNgai runs no persistent network
  service. Path: Settings → พื้นที่ทำงาน → "เชื่อมต่อ Google Drive" (Pro only).
- com.apple.security.network.client — outbound Google Drive API calls that
  upload and download the user's own synced data. Same path as above, Pro only.
- com.apple.security.application-groups (79QFYKTJMN.com.visarut.billngai) —
  required for Chromium's inter-process Mach port rendezvous under the sandbox.
  Without it the app opens to a blank window (we hit exactly this in TestFlight
  build 2.0.3). It backs the app's ability to render at all, so it is exercised
  on every launch.
- com.apple.security.files.user-selected.read-write — two uses, both available
  without Pro. (a) Reading image files the user picks to brand their documents:
  Settings (gear) → "ธุรกิจของฉัน" (My business) → the card "โลโก้ · ลายเซ็น ·
  ตราประทับ" (Logo · Signature · Stamp) → any "เลือกรูป" (Choose image) button,
  which opens the standard file picker. The same picker appears in step 3 of the
  first-run setup wizard. (b) Writing files the user chooses a location for:
  open any document and use "บันทึกเป็น PDF" (Save as PDF), or Settings →
  "ข้อมูลและสำรอง" (Data & backup) → the export/import backup buttons, each of
  which opens a standard save/open panel.

- com.apple.security.cs.allow-jit — required by the V8 JavaScript engine in the
  Electron/Chromium runtime the app is built on. Without it the app crashes at
  launch with EXC_BREAKPOINT in v8::Isolate::Initialize (TestFlight build
  2.0.2). Exercised on every launch.
- com.apple.security.cs.allow-unsigned-executable-memory — required by the same
  V8/Chromium runtime for the same reason, and fails the same way without it.
  Exercised on every launch.

We are not requesting any entitlement we do not use.
